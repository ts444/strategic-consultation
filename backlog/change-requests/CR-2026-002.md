# CR-2026-002: Templates should warn authors not to use claim-label names in bracketed prose

## Problem

During the Meridian Logistics engagement, `01-situation/situation.md` included descriptive prose that used claim-label names inside square brackets as part of explanatory text — for example, "data points should carry assumed labels" or "with elicited labels in brackets". The `label_producer_binding` validator rule uses a regex that pattern-matches any bracketed token matching a claim-label name anywhere in the document body, regardless of context.

This caused false-positive violations: the synthesizer was flagged for using labels not permitted by its producer binding, even though the bracketed terms appeared in descriptive context, not as actual claim annotations. Fixing required a validator rerun after rephrasing the prose.

The same class of issue could arise in any template that uses bracketed label names in instructional prose or example text outside of comment blocks.

## Proposed Change

Add a visible author warning callout to the following templates (outside HTML comment blocks, so it is visible in the rendered document and cannot be inadvertently triggered):

- `templates/situation.template.md`
- `templates/retro.template.md`

The callout should read approximately:

> **Author note:** Do not write claim-label names (`known`, `inferred`, `elicited`, `assumed`) inside square brackets in descriptive prose. The validator treats any such token as a claim annotation and will flag it if the label type does not match the artifact's `produced_by` frontmatter field. Use plain text instead (e.g., "inferred label" not the bracketed form).

As a longer-term alternative, the validator's `label_producer_binding` rule could be tightened to only flag a label token when it is followed by a source annotation on the same line — which is the actual claim-atom pattern. This would be a validator change request and is left for a future CR.

## Expected Improvement

Authors producing situation and retro documents will be less likely to trigger `label_producer_binding` false positives from descriptive prose. The first-run validator pass becomes more likely to succeed without reruns.

## Scope

- `templates/situation.template.md` — add author callout
- `templates/retro.template.md` — add author callout
